# Remediation Exports -- Sample Output

Generated by `Export-M365Remediation` (D3 #787) from a synthetic 3-finding assessment (1 Fail, 1 Warning, 1 Review).

## What's here

| Path | Format | Purpose |
|---|---|---|
| `executive-summary.md` | Markdown | 1-page overview for non-technical stakeholders |
| `jira-import.csv` | CSV | Importable via Jira's CSV importer (Summary, Description, Issue Type, Priority, Labels, Components) |
| `technical-backlog.md` | Markdown table | Full per-finding backlog grouped by horizon (Now / Next / Later) |
| `github/<CheckId>.md` | Markdown (one per finding) | Ready for `gh issue create -F` |
| `github/create-issues.sh` | Shell | Bulk-create GitHub issues from the .md files |

## Generating your own

```powershell
# All four formats (default)
Export-M365Remediation -AssessmentFolder ./M365-Assessment/Assessment_20260426_133012_contoso

# Just the formats your workflow needs
Export-M365Remediation -AssessmentFolder ./M365-Assessment/Assessment_20260426 `
    -Format GitHub, Jira -TenantName 'Contoso'
```

The cmdlet does NOT re-run the assessment -- it reads the existing per-collector CSVs and the registry-driven horizon (Now/Next/Later) the same way the HTML report and XLSX matrix do. So the exports stay in agreement with the rest of the report.

See [`docs/SCORING.md`](../../SCORING.md) for the related "Quick Wins" view in the HTML report's executive-summary tabs.
